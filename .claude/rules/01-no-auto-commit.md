# Rule 01 — 자동 커밋 금지 / 커밋 메시지 포맷

## 룰
1. 작업 및 테스트가 완료되더라도 `git commit`을 직접 실행하지 않는다.
   대신 **커밋 메시지(안)만** 사용자에게 전달한다.
2. 커밋 메시지(안)를 전달할 때는 **관련 변경 파일을 `git add`(스테이징)까지 함께 수행**한다.
   사용자가 메시지만 복사해 바로 `git commit` 하면 되도록 준비한다.
3. 모든 커밋 메시지는 **한글**로 작성한다.
4. 커밋 메시지 제목은 다음 포맷을 따른다:

```
# [주제] - 커밋 메시지
```

## 포맷 규칙
- `#` 으로 시작 (1자)
- 공백 1칸 후 `[주제]` (대괄호 안에 분류 키워드)
- 공백 1칸 + `-` + 공백 1칸
- 이어서 한 줄 요약 (한글)

## 주제 키워드 (권장)
| 주제 | 사용 시점 |
|------|-----------|
| `[feat]` | 신규 기능 추가 |
| `[fix]` | 버그 수정 |
| `[refactor]` | 동작 변경 없는 구조 개선 |
| `[chore]` | 빌드/설정/메타 파일 등 |
| `[docs]` | 문서/주석/README |
| `[test]` | 테스트 코드 추가/수정 |
| `[style]` | 포매팅, 세미콜론 등 비기능 변경 |
| `[perf]` | 성능 개선 |
| `[asset]` | 에셋/프리팹/씬 변경 |
| `[infra]` | ChvjPackage 인프라 변경 |

## 적용 시점
- 코드 작성/수정/리팩터링 완료 후
- 테스트 작성 및 통과 확인 후
- 사용자가 명시적으로 "커밋해" 라고 요청하기 전

## 동작 방식
1. 변경 내용 요약
2. **관련 변경 파일을 `git add`(스테이징)**
3. 위 포맷의 제안 커밋 메시지를 코드블록으로 전달
4. 사용자가 직접 커밋하거나, 명시적 승인 시에만 커밋 실행

## add 범위 주의
- 관련 변경 파일만 스테이징 (`git add -A` / `git add .` 지양 — 무관 파일·민감 파일 혼입 방지)
- `.env`, 인증 토큰 등 민감 파일은 제외
- `.gitignore` 매칭 항목·외부 도구 산출물은 자연히 제외됨을 확인

## 예시
```
# [feat] - Addressables 로더 풀링 지원 추가

- ResourceManager.LoadAsync 캐시 동작 변경
- 풀 반환 시점 명시
```

```
# [fix] - 인벤토리 슬롯 클릭 시 NRE 수정
```

```
# [infra] - ChvjPackage Pool 모듈에 IPoolable 인터페이스 추가
```

## 금지 예시
```
feat: add pooling          //# (X) 영문 + 콜론 포맷
[feat] add pooling         //# (X) # 누락
# feat - add pooling       //# (X) 대괄호 누락
# [feat] 풀링 추가          //# (X) ` - ` 구분자 누락
```
