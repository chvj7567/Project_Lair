## 2. 차단 원인 — 카드 선택 팝업이 자동 픽 API 를 노출하지 않음

### 2.1 1차 증거 — 자연 트리거 발생 시 전투가 팝업에서 멈춤

`BattleController.TryProcessNext()` (`Assets/_Lair/Scripts/Battle/BattleController.cs:232~274`):

```csharp
_pause.Pause();                                        //# 240줄대 — Time.timeScale=0
var choices = deck.Draw(3);
var tcs = new TaskCompletionSource<bool>();
var arg = new CardSelectionArg { Choices = choices, OnPicked = card => { ... tcs.TrySetResult(true); } };
await CHMUI.Instance.ShowUIAsync(EUI.CardSelectionPopup, arg);
await tcs.Task;                                        //# OnPicked 가 호출될 때까지 영구 블록
_pause.Resume();
```

- 패시브 트리거(HP 10%마다, `PassiveTriggerService`)와 액티브 트리거(30초마다, `ActiveTriggerService`)는
  **시뮬레이션 중에도 정상 발화**한다 — `_queue.Enqueue` → `TryProcessNext` 호출.
- `TryProcessNext` 는 `_pause.Pause()` 로 `Time.timeScale=0` 을 걸고, `CardSelectionPopup` 을 띄운 뒤
  `await tcs.Task` 에서 **무한 대기**한다. `tcs` 는 팝업의 `OnPicked` 콜백이 호출돼야만 풀린다.
- 따라서 어떤 자동 픽 메커니즘이 팝업을 실제로 닫지 않으면, **첫 트리거에서 전투가 영구 정지**한다.

### 2.2 작업 브리프의 "v1" 전제는 불완전함

브리프는 `DebugApplyCard(ECardId)` 만으로 사전 정의 카드 리스트를 적용하는 것이 유효한 v1 이라고 기술한다.
**그러나 이는 성립하지 않는다.** `DebugApplyCard` (`BattleController.cs:337~348`)는 카드 *효과*를
즉시 적용할 뿐, 큐에 쌓인 트리거나 화면에 떠 있는 팝업을 해소하지 못한다.
스케줄에 맞춰 `DebugApplyCard` 를 호출해도 §2.1 의 팝업 블록은 그대로 남아 전투가 멈춘다.

### 2.3 팝업 자체에 자동 픽 진입점이 없음

`CardSelectionPopup` (`Assets/_Lair/Scripts/UI/CardSelectionPopup.cs`):

- `CardSelectionArg.Choices` (3장)와 `OnPicked` 콜백은 `InitUI` 의 클로저 안에서만 사용된다.
- 픽은 오직 `CardView.Bind` → `CHButton.OnClick` 으로 등록된 버튼 클릭 핸들러로만 발생한다.
- 팝업에 `PickByIndex(int)` / `Pick(CardData)` 같은 public 메서드, 또는 `Choices` 접근자가 **없다**.
- `CardView` 도 바인딩된 `CardData` 와 `_pickButton` 을 모두 private serialized 필드로 숨긴다.

→ 하베스가 "지금 제시된 3장"을 보거나 그중 1장을 고를 공개 경로가 전혀 없다.

### 2.4 리플렉션 경로는 채택 불가

기술적으로는 라이브 전투의 `CardSelectionPopup._slots`(private) → 각 `CardView._pickButton`(private) →
`Button.onClick.Invoke()` 를 리플렉션으로 호출하는 것이 가능하다. 하지만 이 경로는 거부한다:

- **Rule 03(종속성 최소화) 정면 위반** — 다른 어셈블리의 private serialized 필드 2계층을 파고드는 것은
  최대 결합 안티패턴. UI 가 조금만 재구성돼도 조용히 깨지는 시뮬레이터는 "인프라"가 아니다.
- 전략이 "제시된 3장을 보고 고르는" 동작을 하려면 카드 정체성(`CardData`)이 필요한데, 이 역시
  리플렉션으로만 얻을 수 있어 브리프의 하드 제약("게임 로직을 건드리지 않는다")을 우회하는 꼴이 된다.

## 3. 요청 — gameplay-programmer 에게 필요한 최소 훅

`BattleController` 에 `#if UNITY_EDITOR` 로 감싼 델리게이트 1개 추가. 약 10줄 변경.

```csharp
#if UNITY_EDITOR
//# 비-null 이면 카드 팝업 대신 호출되어 즉시 픽을 결정. 시뮬레이션 전용.
//# 인자: 제시된 3장 choices, 트리거 출처(Passive/Active). 반환: 고른 카드(null 이면 스킵).
public System.Func<IReadOnlyList<CardData>, TriggerQueue.Source, CardData> DebugAutoPicker;
#endif
```

`TryProcessNext()` 의 `var choices = deck.Draw(3);` 직후에 분기:

```csharp
#if UNITY_EDITOR
if (DebugAutoPicker != null)
{
    var picked = DebugAutoPicker(choices, entry.SourceType);
    if (picked != null)
    {
        _recorder.RecordPick(picked.Id);
        _vm.AddPick(picked, entry.SourceType == TriggerQueue.Source.Passive);
        picked.Effect?.Apply(_ctx);
    }
    continue;   //# 팝업/일시정지 스킵 — tcs 블록 회피
}
#endif
```

**이 훅의 장점:**
- 프로덕션 경로 불변 — `#if UNITY_EDITOR` + null 기본값. 일반 플레이는 영향 0.
- 하베스 전략이 "실제 제시된 3장"을 인자로 받음 → 사전 정의 리스트가 아닌 **진짜 전략**(제시된 카드 평가) 구현 가능.
- 팝업/일시정지를 건너뛰어 §2.1 의 `tcs` 무한 대기를 원천 회피.
- `RunRecorder.RecordPick` / `BattleViewModel.AddPick` 호출을 유지해 기존 기록 파이프라인 그대로 재사용.

> **대안(더 작지만 덜 유연):** `BattleController` 에 `EDebugPickStrategy` enum 드롭다운을 두고
> 내장 전략(Random/Tanker/Dealer/AoE)을 골라 쓰는 방식. 델리게이트보다 작지만 전략이 게임 코드에 박힌다.
> 어느 쪽을 택할지는 game-designer / 사용자가 결정. qa-simulator 권장은 델리게이트(유연성).

## 4. 훅이 들어오면 즉시 구축할 하베스 설계 (참고)

게임 로직 훅이 확정·구현되면 qa-simulator 가 다음을 **시뮬레이션 인프라로만** 작성한다 (게임 로직 무수정):

| 구성요소 | 위치(예정) | 역할 |
|---|---|---|
| `LairSimWindow` (EditorWindow) | `Assets/_Lair/Editor/` | 전략 선택 + N 입력 + 실행/진행률. `LairBalanceWindow` 자동화 대응물 |
| `SimDriver` (MonoBehaviour) | `Assets/_Lair/Editor/` 또는 별도 sim asmdef | Play 진입 시 씬에 주입. `BattleController.DebugAutoPicker` 에 전략 픽 함수 연결, 판 종료 감지 후 다음 판 |
| 전략 픽 함수 | `SimDriver` 내부 | `Func<choices, source, CardData>` 4종 — Random / TankerPriority / DealerPriority / AoEPriority |

- **틱**: Play 모드 + `Time.timeScale` 가속(15~20x → 5분 전투를 ~15~20초). 순수 모델 시뮬은 불가
  (`BattleController.Start` 가 `async void` + `CHMResource.Init`/`Addressables`/`CHMPool`/`CHMUI` 의존, 전투 AI·Health·Attacker 가 MonoBehaviour — 분리하려면 게임 코드 대규모 리팩터 필요).
- **도메인 리로드 생존**: `LairTestRunner` 의 `SessionState` 패턴을 미러링 (Play 진입/종료 시 2회 리로드).
- **메트릭** (컨셉 §8 기준):
  - 평균 영웅 사망 시각 — 목표 2~4분
  - 클리어율 (5분 내 처치 비율)
  - 카드별 픽률 / 픽 시 승률
  - 빌드 다양성 — 지배 시너지
  - 사망 시점 분포 / 종료 시 생존 몬스터 수
- 데이터 소스는 기존 `RunRecorder` jsonl(`Logs/lair_runs.jsonl`) 재사용 — 픽·결과·사망시각·생존수가 이미 기록됨.
