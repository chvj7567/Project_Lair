## 1. 레퍼런스 메커니즘 분석

### 1.1 동작

Vampire Survivors 의 패시브 시너지 시스템은 세 층위로 구성된다. 첫 번째는 **아이템–무기 진화 조건**: 각 무기는 특정 패시브 아이템을 특정 레벨까지 보유하면 황금 보석 습득 시 업그레이드된 '진화 무기'로 자동 교체된다(예: Magic Wand + Empty Tome → Holy Wand). 두 번째는 **제안 가중치 알고리즘**: 레벨업 선택지 3장은 완전 무작위가 아니라 ① 현재 보유한 무기/아이템의 업그레이드 선택지를 우대하고 ② 아직 없는 아이템에도 기회를 열어 빌드 폭을 확보하도록 가중된다. 세 번째는 **암묵적 시너지 단서**: 레벨업 창에 진화 가능 무기의 반짝이는 테두리 효과가 나타나 "이 아이템을 집으면 무언가 달라진다"는 신호를 주되, 정확한 조합법은 플레이어가 스스로 발견하도록 남겨둔다. 아이템 설명 문구에 "can be combined with [weapon name]" 같은 단서가 짧게 삽입되어 있어 메타 지식 없이도 2~3판이면 첫 진화를 체험할 수 있다.

### 1.2 왜 잘 될까

시스템이 잘 작동하는 이유는 **탐색 부담을 게임 루프 안에 녹여냈기** 때문이다. 플레이어는 진화 공식을 외울 필요 없이 "빛나는 무기가 있으면 관련 아이템을 집으면 된다"는 단순 규칙만 익히면 첫 진화를 경험할 수 있다. 진화 순간은 짧지만 강렬한 시각·사운드 연출로 '보상 클라이맥스'를 제공하여, 그 조합을 찾아냈다는 숙달감(mastery)을 확정적으로 준다. 또한 제안 알고리즘이 보유 아이템 업그레이드를 우대하기 때문에 막 진화 조건을 충족하려는 빌드 경로가 "도중에 끊기는" 좌절이 적다 — 선택지 3장 중 적어도 1장은 현재 빌드와 연관된 제안이 나와 결정의 의미가 유지된다.

### 1.3 제약·한계

이 시스템은 아이템 풀이 40~80종 이상으로 커야 충분히 작동한다. 풀이 작으면 가중치 알고리즘을 써도 선택지 다양성이 좁아 "어차피 나오는 게 뻔하다"는 인상을 준다. 또한 진화 조합법 수가 증가할수록 메타 지식 의존이 심해지고 신규 플레이어 진입 장벽이 올라간다. VS 후기 버전이 인게임 진화 레시피 표를 추가한 것도 이 때문이다. Project Lair 는 MVP 에서 카드 풀이 25장(패시브 15 + 액티브 10)으로 작으므로 VS 의 진화 조합보다는 **태그 기반 시너지 시그널**과 **가중치 제안**에 집중하는 것이 현실적이다.

## 2. Project Lair 적용 후보 (패턴 3개)

### 2.1 패턴 1: 빌드 태그 가중 제안 (Build-Tag Weighted Offering)

- **적용 대상**: `CardData` ScriptableObject + `CardDeck.Draw()` 또는 신규 `CardOfferingService`
- **구현 스케치**: `CardData` 에 `BuildTag[]` 필드(예: `[Undead, Slow, Ranged]`)를 추가하고, 플레이어가 픽한 카드의 태그 빈도를 `CardOfferingService` 가 추적한다. `Draw(3)` 호출 시 가장 많이 집은 태그와 일치하는 카드에 가중치(예: 기본 1.0 → 2.0)를 부여하여 후보 풀을 샘플링한다. 가중치는 픽 횟수가 늘수록 올라가되 풀 내 다른 태그 카드도 최소 1장 포함되도록 보장(다양성 보존 규칙)한다. ScriptableObject 에 필드 하나 추가 + 가중 랜덤 샘플링 로직이 전부이므로 구현량이 적다.
- **구현 비용**: 2
- **MVP §11 호환**: O — 카드 풀 15+10장에서 동작. 아트 변경 없음.
- **적재 타이밍**: Alpha (B3 컨텐츠 확장 직후 — 카드 풀이 25장 채워진 시점)
- **의존 패턴**: B1/B2 에서 이미 완성된 `CardData` SO 스키마, `CardDeck.Draw(n)` 인터페이스

---

### 2.2 패턴 2: 시너지 임박 시그널 (Synergy Proximity Signal)

- **적용 대상**: `CardSelectionPopup` (View) + `CardView` + `CardData`
- **구현 스케치**: 카드 선택 팝업이 뜰 때, 후보 카드 중 플레이어의 현재 빌드 태그 세트와 **1장만 더 있으면 시너지 임계치**를 넘는 카드에 시각 단서를 표시한다 — 예: 카드 테두리에 별도 컬러 펄스 효과 또는 "시너지 가능" 아이콘(프리미티브 단계에서는 카드 배경색 변경). VS 의 진화 무기 반짝임과 동일한 원리로, 조합 공식을 텍스트로 설명하지 않고 시각 단서만 노출해 호기심을 유발한다. `CardSelectionArg` 에 `SynergyHighlight: HashSet<ECardId>` 를 추가하고, `CardView` 는 해당 ID 면 하이라이트 상태로 렌더링한다.
- **구현 비용**: 2
- **MVP §11 호환**: O — UI 프리미티브 색상 변경만. 아트 불필요.
- **적재 타이밍**: Alpha
- **의존 패턴**: 패턴 1 의 BuildTag 필드 + 시너지 임계치 정의 (game-designer 가 §5.2 기준으로 작성)

---

### 2.3 패턴 3: 픽 이력 기반 바닐라 보호 (Pick-History Diversity Guard)

- **적용 대상**: `CardOfferingService` (패턴 1 의 서비스 확장)
- **구현 스케치**: 플레이어가 같은 카드를 이미 픽했거나, 같은 태그 카드를 연속 3번 이상 픽한 경우 다음 제안에서 그 카드 / 태그를 한 회 강제 억제한다("ban slot"). VS 의 "이미 맥스 레벨이라 제안 안 됨" 로직과 동일한 원리. 25장 풀에서 연속 같은 태그만 나오는 루트를 방지하고, 플레이어가 새로운 시너지 경로를 탐색할 기회를 보존한다. `CardOfferingService` 의 `recentPicks` 링 버퍼(크기 3)를 보고 샘플링 전에 억제 태그를 제외하는 간단한 필터로 구현.
- **구현 비용**: 1
- **MVP §11 호환**: O
- **적재 타이밍**: Alpha (패턴 1 과 동시 구현 권장 — 같은 서비스 내 필터로 추가)
- **의존 패턴**: 패턴 1 의 `CardOfferingService` + `recentPicks` 추적 구조

## 3. 우선순위 제안

**패턴 1 → 패턴 3 → 패턴 2** 순서로 구현 권장.

패턴 1(가중 제안)은 플레이어가 빌드 방향을 잡을 수 있는 가장 직접적인 레버이며 모든 카드 콘텐츠가 지원된다. 패턴 3(다양성 보호)은 패턴 1 과 동일 서비스에 구현하므로 추가 비용이 거의 없고 단조로운 제안 문제를 즉시 예방한다. 패턴 2(시너지 시그널)는 시너지 임계치 수치 정의가 game-designer 작업을 선행해야 하므로 마지막 단계에 위치한다. 세 패턴 모두 MVP §11 범위 안이지만, 현재 B3 컨텐츠 확장이 완료되지 않은 상태에서는 Alpha 단계에서 일괄 구현하는 것이 가장 낭비 없는 타이밍이다.

## 4. 채택 흐름 제안

- **패턴 1+3**: B3 컨텐츠 확장(카드 풀 25장 완성) 직후 → game-designer 에게 이 문서 전달 + BuildTag 정의 요청 → gameplay-programmer 가 `CardData` 스키마 확장 + `CardOfferingService` 구현
- **패턴 2**: 시너지 임계치 수치가 game-designer 기획서에 확정된 이후 → `CardSelectionArg`, `CardView` 하이라이트 추가
- **v0.2 이후 고려**: 카드 풀이 60장+ 으로 늘어나면 VS 의 진화 카드 조합(레시피 기반 강화) 도입 검토. 현 MVP 25장에서는 태그 수준이 충분하며 레시피는 과설계.
